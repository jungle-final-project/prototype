# BuildGraph 성능 테스트 전체 계획

## 1. 문서 목적

이 문서는 현재 코드와 2026-07-12 부품 목록 1차 부하 테스트 결과를 기준으로, 이후 성능 테스트의 대상·순서·판정 기준·중단 기준·결과 저장 방식을 확정한다.

이번 단계는 계획 확정 단계다. 애플리케이션 코드나 k6 코드를 변경하지 않고 실제 부하도 실행하지 않는다. 이후 각 단계는 이전 단계의 검증·커밋·push가 끝난 뒤 별도 작업으로 진행한다.

## 3. 테스트 환경

### 3.1 현재 확인된 구성

| 영역 | 현재 구성 |
|---|---|
| 호스트 | Windows + Docker Desktop |
| Docker | Engine 29.2.1, Compose v5.1.0 |
| API | Spring Boot 3.4.1, Java 21 |
| DB | `pgvector/pgvector:pg16` 기반 PostgreSQL 16 |
| 부하 도구 | Docker `grafana/k6:0.54.0` |
| Compose 프로젝트 | `prototype` |
| API 컨테이너 | `buildgraph-api` |
| DB 컨테이너/볼륨 | `buildgraph-postgres` / `prototype_postgres-data` |
| 기타 서비스 | Redis, RabbitMQ, Mailpit, XGBoost reranker, Web |

### 3.2 Actuator와 Micrometer

- `spring-boot-starter-actuator`가 의존성에 포함되어 있다.
- 기본 `application.yml`은 `health,info`만 노출한다.
- 현재 working tree의 `application-loadtest.yml`은 `health,info,metrics`와 Tomcat MBean registry를 켜도록 작성돼 있다.
- 현재 실행 중인 API는 기본 프로필이며 `/actuator/metrics`가 HTTP 404다.
- Prometheus registry 의존성은 없다. 2단계에서는 Prometheus/Grafana 전체 스택보다 Actuator JSON 표본을 우선한다.
- 실제 metric 이름은 2단계에서 `loadtest` 프로필 적용 후 `/actuator/metrics` 목록으로 확정한다. 존재하지 않는 이름을 미리 하드코딩하지 않는다.

### 3.3 HikariCP와 PostgreSQL

- datasource는 Spring Boot 자동 구성을 사용하고 HikariCP 풀 크기·timeout의 명시적 프로젝트 설정은 없다.
- PostgreSQL Compose에는 `shared_preload_libraries=pg_stat_statements` 설정이 없다.
- 현재 활성 extension은 `pgcrypto`, `plpgsql`, `vector`이며 `pg_stat_statements`는 활성화되어 있지 않다.
- `pg_stat_statements` 활성화에 DB 재시작이 필요하면 자동 적용하지 않고 별도 승인을 받는다.
- 기존 `prototype_postgres-data` 볼륨은 삭제하거나 초기화하지 않는다.

### 3.4 실행 환경 고정 항목

각 실행의 `environment.json`과 `manifest.json`에 다음을 기록한다.

- KST 시작·종료 시각
- branch, commit, dirty 여부와 변경 파일 목록
- 호스트 CPU·메모리·OS
- Docker·Compose·k6 버전
- 컨테이너 이미지 ID와 컨테이너 자원 제한
- 활성 Spring profile
- DB 데이터 기준 또는 식별 가능한 fixture 버전
- BASE_URL, API 파라미터, VU, duration, think time
- 테스트 계정/fixture 식별자(비밀번호·token 제외)

## 4. 대표 기능 선정

모든 대표 API는 JWT Bearer 인증이 필요하다. 토큰은 `POST /api/auth/login`으로 발급한다.

### A. 읽기 중심: 부품 목록

```http
GET /api/parts?category=CPU&page=0&size=20
```

선정 이유:

- 프론트의 부품 목록·검색과 셀프견적 후보 조회에서 사용하는 핵심 읽기 경로다.
- 페이지네이션된 DB 조회이며 외부 쇼핑 API를 실시간 호출하지 않는다.
- HTTP 200과 `items` 배열로 계약 검증이 명확하다.
- 기존 k6와 1차 결과를 그대로 기준선으로 재사용할 수 있다.

### B. 복합 조회·연산: Build Graph 계산

```http
POST /api/build-graphs/resolve
Content-Type: application/json

{
  "source": "QUOTE_DRAFT_CURRENT",
  "view": "FULL"
}
```

이 endpoint는 HTTP method가 POST지만 DB를 변경하지 않는 계산성 조회다.

선정 이유:

- `quote_drafts`, `quote_draft_items`, `parts`를 읽는다.
- compatibility, power, size, performance, price Tool 판정과 composite score·그래프 node/edge 계산을 수행한다.
- Tool 계산 과정에서 저장된 benchmark/FPS 근거를 조회할 수 있다.
- 셀프견적 화면과 구매 확인 화면에서 실제 호출한다.
- OpenAI, 네이버 쇼핑, 다나와를 실시간 호출하지 않는다.
- 응답의 `mode`, `summary`, `nodes`, `edges`, `focusNodeIds`, `insights`, `toolResults` 계약을 검증할 수 있다.

사전 조건:

- 테스트 계정마다 ACTIVE 견적초안과 호환 가능한 부품 fixture가 필요하다.
- 반복 실행 중 같은 견적 데이터를 유지한다.

### C. 쓰기·트랜잭션: 조립 요청 생성

```http
POST /api/assembly-requests
Authorization: Bearer {accessToken}
Idempotency-Key: PERF-{runId}-{vuId}-{iteration}
Content-Type: application/json
```

선정 이유:

- 프론트의 구매 확인 화면에서 실제 호출한다.
- 현재 견적을 읽고 Build Graph 호환성을 다시 검증한다.
- 한 transaction에서 `assembly_requests`, `assembly_request_items`, 상태 이력, 조건에 맞는 자동 기사 제안을 저장한다.
- `(user_id, idempotency_key)` unique 제약과 request fingerprint 검증이 있어 중복·충돌을 측정할 수 있다.
- 실제 PG 결제나 외부 기사 API를 호출하지 않는다. 결제 endpoint도 현재는 가상 결제 상태 전이만 제공한다.

테스트 데이터 격리:

- 성능 테스트 전용 계정 풀을 사용하고 각 계정에 호환 가능한 ACTIVE 견적초안을 준비한다.
- 모든 `Idempotency-Key`와 `note`에 `PERF-{runId}` 접두사를 넣는다.
- `preferredDate`는 실행일 이후의 고정 날짜 규칙을 사용하고 연락처·주소는 명백한 synthetic 값만 사용한다.
- 요청 성공 수와 `assembly_requests` 레코드 수, 품목 snapshot 수, 상태 이력 수를 비교한다.
- 정리는 자동 수행하지 않는다. 필요한 경우 해당 runId로 생성된 행만 식별하는 검토된 SQL을 별도 승인 후 실행한다.

견적초안 item PUT을 대표 쓰기로 선택하지 않은 이유는 사용자별 현재 draft 한 개를 여러 VU가 동시에 변경하면 동일 행 경합과 마지막 쓰기 승리 때문에 실제 저장 처리량보다 공유 fixture 경쟁을 측정할 가능성이 높기 때문이다.

## 5. 기능별 사용자 행동과 혼합 시나리오 초안

실제 프론트 호출을 기준으로 한 기본 흐름:

```text
로그인
→ 부품 목록 조회
→ 부품 상세 또는 호환 후보 조회
→ 현재 견적 확인 및 전용 계정의 견적 item 선택
→ Build Graph/성능 확인
→ 일부 사용자만 조립 요청 생성
```

실제 트래픽 통계가 없으므로 최초 혼합 비율은 다음 가정으로 시작한다.

| 그룹 | 임시 비율 | 포함 요청 |
|---|---:|---|
| 부품 탐색 | 60% | 목록·검색·상세 |
| 견적·호환성·계산 | 25% | 현재 견적, 호환 후보, Build Graph |
| 인증 | 10% | 로그인 또는 token refresh |
| 쓰기 | 5% | 조립 요청 생성 |

- 위 비율은 운영 통계가 아니라 가정이다.
- 사용자 행동 사이에는 1~3초의 seeded/random think time을 적용하고 반복 실행에서는 같은 분포를 유지한다.
- 모든 요청마다 로그인하지 않는다. setup에서 전용 계정 token pool을 만들고 인증 그룹만 별도로 로그인·refresh를 수행한다.
- 쓰기 VU는 고유 계정 또는 최소한 고유 idempotency key와 request marker를 사용한다.

## 6. 부하 증가와 반복 단계

### 6.1 부품 목록 반복 검증

```text
50 VU × 3회
100 VU × 3회
```

수집기를 먼저 구축한 뒤 1차 테스트의 50 VU p95 상승이 반복되는지 확인한다.

### 6.2 부품 목록 한계 탐색

```text
100 → 200 → 300 → 500 → 700 → 1,000 VU
```

1,000 VU가 안정적이고 서버와 부하 생성기에 여유가 있을 때만 조건부로 진행한다.

```text
1,500 → 2,000 → 3,000 → 5,000 → 7,500 → 10,000 VU
```

첫 저하 또는 실패 후보가 나타나면 마지막 안정·저하 시작·첫 실패 세 구간을 각각 최소 3회 반복한다.

### 6.3 복합 조회와 쓰기 API

각 API별 순서:

```text
Smoke → Warmup → Baseline → 10 → 50 → 100 → 300 → 500 → 1,000 VU
```

경계가 먼저 나타나면 더 높은 부하는 중단하고 마지막 안정·저하·실패 구간을 각각 3회 반복한다.

### 6.4 혼합 사용자 시나리오

```text
100 → 300 → 500 → 700 → 1,000 VU
```

안정적이고 부하 생성기와 서버 모두 여유가 있을 때만 1,500 → 2,000 → 3,000 → 5,000 VU를 조건부로 수행한다.

## 7. 수집할 k6 지표

- `vus`, `vus_max`
- `iterations`, `iteration_duration`
- `http_reqs`, 실제 RPS
- `http_req_duration`의 avg, med, p90, p95, p99, max
- endpoint·기능 그룹별 `http_req_duration`
- `http_req_failed`
- `checks`와 계약별 pass/fail
- `dropped_iterations`
- data sent/received
- setup 로그인 성공과 token 발급 실패
- 쓰기 요청 성공 수와 생성 public id 수

VU, 동시 HTTP 요청 수, RPS, 동시 접속 사용자 수는 같은 값으로 표현하지 않는다.

## 8. 수집할 서버·DB 지표

기본 수집 주기는 5초이며 모든 표본에 KST timestamp와 runId를 넣는다.

### 8.1 Spring Boot/JVM

- process CPU, system CPU
- JVM Heap used/max
- GC pause count/time
- live/peak thread
- Tomcat busy/max thread
- endpoint별 `http.server.requests`

### 8.2 HikariCP

- active, idle, pending, max
- acquire/usage/creation 시간
- connection timeout 또는 acquire 실패

실제 노출 이름은 2단계의 `/actuator/metrics` inventory에서 확정한다.

### 8.3 PostgreSQL

- `pg_stat_activity`: active/idle/waiting connection과 wait event
- `pg_stat_database`: transaction, commit, rollback, block read/hit
- `pg_locks`: granted=false lock wait
- `pg_stat_user_tables`, `pg_stat_user_indexes`: scan·tuple·index 사용 변화
- 주요 테이블의 실행 전후 row count와 쓰기 정합성
- `pg_stat_statements`는 승인 후 활성화된 경우에만 SQL별 calls/mean/total time을 수집한다.

### 8.4 Docker와 부하 생성기

- API·PostgreSQL CPU, memory, PIDS
- Network I/O, Block I/O
- k6 컨테이너/호스트 CPU와 memory
- 컨테이너 restart count와 health

## 9. 임시 판단 기준

공식 SLA가 없으므로 아래는 탐색과 전후 비교용 임시 기준이다.

| 대상 | checks | HTTP 실패율 | p95 | 추가 기준 |
|---|---:|---:|---:|---|
| 부품 목록 | ≥ 99% | < 1% | < 500ms | `items` 계약 오류 0 |
| Build Graph | ≥ 99% | < 1% | < 1,000ms | graph/toolResults 계약 오류 0 |
| 조립 요청 | ≥ 99% | < 1% | < 1,000ms | 데이터 정합성 오류 0 |
| 혼합 시나리오 | ≥ 99% | < 1% | 기능 그룹별 판정 | API·DB health 정상 |

한 단계가 기준을 한 번 넘었다고 즉시 서버 한계로 확정하지 않는다. 동일 구간 반복 결과와 같은 시간대 서버·DB 지표가 함께 악화됐을 때 저하 또는 포화 후보로 기록한다.

## 10. 테스트 중단 기준

다음 중 하나가 반복되거나 지속되면 더 높은 부하로 진행하지 않는다.

- HTTP 실패율 1% 이상
- checks 99% 미만 또는 데이터 계약 오류
- API/DB health DOWN 또는 컨테이너 재시작
- Hikari pending 지속 증가, connection timeout
- DB lock wait 지속 또는 rollback 급증
- API/DB CPU 장시간 포화
- Heap이 회수되지 않거나 GC pause 급증
- Tomcat busy thread가 max에 지속 접근
- `dropped_iterations` 증가
- k6 호스트가 먼저 포화되거나 목표 VU를 생성하지 못함
- 쓰기 성공 수와 생성 레코드 수 불일치, 중복·불완전 레코드

운영 URL, 외부 LLM, 실제 결제 endpoint가 감지되면 실행 전에 중단한다.

## 11. 외부 API 처리 원칙

코드에서 확인한 외부 경계:

- OpenAI Responses/Embedding: AI chat, Agent, RAG embedding과 일부 제조사 asset 생성
- 네이버 쇼핑: offer/catalog refresh
- 다나와: 가격 snapshot/trend refresh
- XGBoost reranker: Compose 내부 HTTP 서비스

원칙:

- 부품 목록, Build Graph, 조립 요청 대표 테스트는 저장된 DB 데이터만 사용하며 외부 OpenAI·쇼핑 API를 호출하지 않는다.
- 대규모 혼합 테스트에서도 LLM·네이버·다나와 호출 endpoint는 제외하거나 명시적 Stub/Mock을 사용한다.
- 실제 OpenAI·쇼핑 API는 별도 소규모 테스트에서 응답시간·실패율만 측정하며 비용·rate limit을 기록한다.
- 실제 결제 부하는 수행하지 않는다. 현재 가상 결제 상태 전이도 별도 쓰기 시나리오로 분리한다.
- 스케줄러가 외부 호출과 결과를 교란하지 않도록 2단계에서 테스트 profile의 scheduler 설정을 확인하되, 임의로 운영 설정을 바꾸지 않는다.

## 12. 결과 저장 구조

원시 결과는 Git에서 제외하고 요약 문서만 commit한다.

```text
infra/k6/reports/performance/
└─ {yyyyMMddTHHmmssKST}-{gitCommit}/
   ├─ environment.json
   ├─ manifest.json
   ├─ parts-list/
   │  ├─ 50vu-run1/
   │  ├─ 50vu-run2/
   │  ├─ 50vu-run3/
   │  ├─ 100vu-run1/
   │  └─ ...
   ├─ build-graph/
   ├─ assembly-request/
   └─ mixed/
```

각 실행 디렉터리:

```text
k6-console.log
k6-summary.json
k6-report.html          # 생성 도구를 추가할 때만
spring-jvm.csv
hikari.csv
postgres.csv
docker-stats.csv
run.json
```

보호 규칙:

- 기존 결과 디렉터리를 덮어쓰지 않는다.
- 운영 URL을 차단한다.
- health DOWN이면 실행하지 않는다.
- 기존 컨테이너·볼륨·DB 데이터를 자동 삭제하지 않는다.
- 실패해도 원시 결과를 보존한다.
- secret, password, access token을 파일에 저장하지 않는다.
