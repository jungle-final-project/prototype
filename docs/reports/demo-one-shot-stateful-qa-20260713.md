# 데모 원샷 상태 기반 QA 보고서

## 1. 목적과 결론

기존 Build Chat QA는 개별 API 요청의 schema, 예산 밴드, Tool FAIL 제외, draft 무변조를 폭넓게 검증했다. 그러나 실제 사용자는 한 문장으로 끝나지 않고 추천을 적용한 뒤 부품을 바꾸고, 이전 추천을 다시 선택하고, 구매·기사·AS 화면으로 이동한다. 이 보고서는 그 연속 상태를 실제 Docker 웹, API, DB에서 한 번에 따라가며 기존 QA가 놓친 오류와 품질 저하를 찾은 결과다.

결론은 다음과 같다.

- 200만원 QHD 추천, GPU 하향·복구, 조립 요청·외부 기사 입찰·결제, 원격지원 상태 갱신의 큰 흐름은 동작한다.
- 하지만 추천 카드의 `균형/성능 통과` 판단과 1000점 평가 조언이 서로 충돌한다.
- 과거 변경 미리보기가 현재 견적보다 강한 문맥으로 재사용되어, 실제로 제거한 RAM을 "이미 들어 있음"으로 오판한다.
- 점수 조언의 후속 칩이 실질적으로 상위 부품을 고르지 못하고 동급 부품을 다시 추천한다.
- AS 로그의 실제 근거는 DB에 존재하지만 발표에서 필요한 형태로 사용자 화면에 노출되지 않는다.
- 따라서 기존의 단일 턴 시나리오 개수만 늘려서는 같은 유형을 잡기 어렵다. 이후 QA의 기본 단위는 `문장`이 아니라 `상태 전이 시나리오`여야 한다.

## 3. 원샷 시나리오 결과

### 3.1 쇼핑·AI 견적

1. `200만 원 예산으로 QHD 게임용 PC 추천해줘`를 입력했다.
2. 1,949,940원, 2,133,560원, 2,193,700원의 세 조합이 반환되어 TARGET 예산 범위 175만~225만원은 지켰다.
3. 가운데 조합을 적용하자 8개 필수 카테고리와 Tool FAIL 없는 견적이 생성됐다.
4. `GPU를 한 단계 낮은 더 저렴한 제품으로 바꿔줘`는 가격, FPS, 종합 점수 변화를 포함한 미리보기를 반환했고 적용도 성공했다.
5. 원래 GPU의 전체 상품명으로 복구 미리보기를 만들고 적용하자 원상 복구됐다.
6. 배치도 위치 질문과 성능 시뮬레이션은 견적을 변경하지 않았다.

### 3.2 수동 견적

신규 사용자로 0/8 상태에서 CPU, 메인보드, RAM, GPU, SSD, PSU, 케이스, 쿨러를 순서대로 직접 선택했다. 알려진 호환 조합은 8/8까지 완성되고 호환성 검사도 통과했다. RAM 수량을 1에서 0으로 낮추면 실제 항목 삭제로 연결되는 것도 확인했다.

### 3.3 조립 중개

1. 선택 연락처와 주소를 비워 둔 상태에서 조립 요청을 생성했다.
2. 내부 제안 2건이 생성됐다.
3. 승인된 외부 기사 계정이 같은 요청에 제안을 제출했다.
4. 사용자 화면은 polling 후 `새 제안 도착`과 내부 2건·외부 1건을 표시했다.
5. `/my/quotes`의 진행 중 의뢰에서 제안 3건 확인 동선이 노출됐다.
6. 외부 제안 선택과 가상 결제가 성공했다.
7. 결제 전에는 기사에게 개인정보가 보이지 않았고, 결제 후 선택된 기사에게만 연락 정보 영역이 표시됐다.

### 3.4 AS 원격지원

`remote_driver_display_tdr.jsonl` fixture로 티켓을 생성하고 원격지원 요청, 관리자 승인, 링크 발급, 해결 메모 저장, 티켓 종료까지 수행했다. 사용자 화면은 약 6초 안에 `LINK_SENT`와 관리자 메모를 갱신했다. 다만 로그 근거 표시 품질은 4.4의 문제로 발표 기준을 충족하지 못했다.

### 3.5 부품 추천 상태형 체인 추가 검증

이번 추가 검증은 단일 응답의 후보 수가 아니라 다음 전체 체인을 8개 카테고리에서 실행했다.

`상품 미지정 담기 요청 → 조건 되묻기 → 서버가 만든 조건 칩 재전송 → 상품 칩 재전송 → 변경 미리보기 → 실제 상태 변경 → 같은 조건 재추천`

- CPU, 메인보드, RAM, GPU, SSD, 파워, 케이스, 쿨러 모두 조건 되묻기와 상품 미리보기까지 연결됐다.
- 상품 칩 재전송은 7~11ms에 단일 미리보기를 만들었고 Tool blocking FAIL은 0건이었다.
- 빈 견적에 RAM을 실제 추가한 뒤 같은 조건으로 다시 추천했을 때 방금 추가한 상품은 제외되고 대체 후보 3개가 채워졌다.
- RAM 슬롯이 이미 찬 견적에서는 억지로 제품 3개를 만들지 않고 현재 RAM 제거·호환 문제 설명 흐름으로 전환했다.
- 단순 `가성비/최저가/고성능 + 카테고리` 추천은 LLM을 거치지 않는 결정적 경로로 옮긴 뒤 실제 API 8개가 12~22ms에 응답했다.

이 과정에서 기존 단건 QA가 놓친 두 결함을 추가로 발견했다. 짧은 clarification 답변 `32GB`가 `추천해줘 32GB`라는 상품명으로 오인됐고, 서버가 스스로 만든 `가성비` 칩은 CPU 9950X3D·GPU RTX 5090·1500W 파워처럼 반대 방향의 고가 제품을 반환했다. 두 문제 모두 아래 수정 항목에 반영했다.

## 4. 발견 사항

### P1-6. 생성된 조건 칩과 후속 답변이 원래 의도를 보존하지 못함 — 수정 완료

**재현 1**

`램 하나 넣어줘` → `32GB`로 답하면 합성 메시지의 `추천해줘 32GB`를 모델명으로 오인해 실제 32GB 자산이 있는데도 `PART_CONSTRAINT_NOT_FOUND`로 끝났다.

**재현 2**

`CPU/GPU/파워 하나 넣어줘`가 만든 `가성비로 추천해줘` 칩을 누르면 각각 9950X3D급, RTX 5090, 1500W 파워가 먼저 나왔다. 칩 문구와 후보 정렬 정책이 반대였다.

**수정**

- 추천·검색 동사와 가격 수식어가 붙은 숫자 스펙을 상품 모델명으로 해석하지 않도록 파서를 보정했다.
- 완전한 후속 명령은 이전 원문과 중복 합성하지 않고, `32GB` 같은 짧은 답만 원 카테고리 문맥과 합성한다.
- 가성비는 CPU/GPU의 벤치마크 대비 가격, RAM/SSD/파워의 대표 정량 스펙 대비 가격으로 정렬한다.
- 객관적 성능 근거가 없는 메인보드/케이스/쿨러는 가짜 성능 점수를 만들지 않고 낮은 가격과 Tool 통과 여부를 사용한다.
- 단순 방향성 부품 추천은 LLM을 생략해 실제 Docker API 기준 12~22ms로 단축했다.
- 응답 의미 변경 후 구 Redis·semantic 응답이 재생되지 않도록 캐시 계약 버전을 갱신했다.

### P1-7. 추천 후 현재 상태와 후보가 다시 충돌함 — 수정 완료

- 현재 견적에 이미 선택된 정확한 RAM/SSD는 ADD 추천 TOP3에서 제외한다.
- 후보 앞부분이 Tool FAIL이어도 최대 50개 범위에서 통과 후보 3개를 채우며, 3개가 모이면 즉시 검사를 중단한다.
- `장착 여유 큰 케이스`는 현재 케이스보다 알려진 GPU·쿨러·파워 여유가 모두 나빠지지 않고 하나 이상 실제로 개선되는 후보만 반환한다.
- 현재 케이스와 같은 수치의 파생 상품, 더 작은 케이스, Tool blocking FAIL 후보는 제외한다.
- 셀프견적 챗봇은 추천 문장 자체에 변경 동사가 없어도 항상 최신 draft를 전송한다. 생성된 칩과 다음 추천이 과거 추천 카드가 아니라 방금 적용한 현재 상태를 기준으로 이어진다.

### P1-1. 추천 카드와 종합 점수 조언이 서로 모순됨

> 2026-07-14 수정: 가격 인덱스 기반 `relabelTierBuilds`를 제거하고 추천 카드도 `BuildEvaluationService`의 공통 1000점 평가 snapshot으로 tier·label·badge를 생성한다. target 예산 카드의 Tool 예산도 사용자 명시 예산으로 재검증한다.

**재현**

- 입력: `200만 원 예산으로 QHD 게임용 PC 추천해줘`
- 가운데 추천 카드: `균형`, 성능 통과, 전력 통과
- 같은 구성을 1000점 평가: 786점 B, `CPU_GPU_IMBALANCE`, `LOW_POWER_HEADROOM`, GPU 및 PSU 상향 우선

**실제 문제**

세 조합은 GPU, RAM, PSU, 케이스가 같고 CPU 위주로만 달라졌다. 가운데와 최고가 조합의 종합 점수는 모두 786점으로 같지만, 가격 순서만으로 각각 `균형`, `고성능` 라벨이 붙었다. 최고가 조합은 BuildGraph 재평가에서 가격 Tool FAIL도 발생했다.

**기대 결과**

- `균형/고성능` 라벨은 가격 위치가 아니라 종합 평가 결과와 실제 성능 차이에 근거해야 한다.
- 추천 카드와 종합 점수 설명은 동일한 Tool 결과, warning code, score를 사용해야 한다.
- 같은 점수와 사실상 같은 게임 성능이면 별도 성능 티어로 과장하지 않아야 한다.

**추정 원인**

- `BuildChatService.nearBudgetLadderBuilds`는 총액 오름차순으로 정렬한 뒤 `relabelTierBuilds`에서 인덱스만으로 value/balanced/performance를 붙인다.
- 추천 생성 Tool 예산과 BuildGraph의 사용 예산 해석이 달라 같은 구성의 가격 판정이 달라질 수 있다.

관련 코드: `BuildChatService.java:3397`, `BuildChatService.java:3694`, `BuildChatService.java:3756`.

### P1-2. 점수 조언의 후속 추천이 문제를 개선하지 않음

**재현**

1. 점수 설명에서 `CPU 체급 대비 GPU가 낮다`는 주의가 표시됐다.
2. 후속 칩 `현재 견적에 맞는 상위 GPU 추천해줘`를 전송했다.
3. 현재 ASUS RTX 5060 Ti 대신 MSI RTX 5060 Ti를 추천했다.

**실제 결과**

- 가격: +14,040원
- 성능 변화: 0%
- 예상 FPS: 140 → 140
- 종합 점수: 786 → 786
- 답변에는 한국어 문장 안에 `विकल्प`이라는 다른 언어 단어도 섞였다.

**기대 결과**

- `상위` 요청은 현재 GPU보다 성능 tier/rank가 확실히 높은 후보만 허용해야 한다.
- 실제 개선 후보가 없으면 동급 교체를 상위라고 부르지 말고 후보 없음 또는 조건 완화를 안내해야 한다.
- 사용자 문장은 한국어 허용 문자와 제품명 외 언어 오염을 후처리해야 한다.

**추정 원인**

`PartReplacementRanker.higher` 자체는 rank 상승을 요구하지만, 점수 조언 후속 문구가 기존 변경 fast path의 명확한 `MORE_EXPENSIVE`로 고정되지 않거나 LLM 구조화 결과가 다른 방향으로 들어갈 수 있다. 결과 카드의 delta를 다시 검사하는 최종 `실질 개선 guard`가 없어 동급 후보가 서빙된다.

관련 코드: `PartReplacementRanker.java:128`, `DefaultAiChatEngine.java:454`, `DefaultAiChatEngine.java:1441`.

### P1-3. 과거 미리보기가 현재 견적을 덮어쓰는 대화 상태 오염

**재현**

1. RAM을 실제 견적에서 제거해 8/8 → 7/8로 만들었다.
2. `Corsair Vengeance RGB DDR5-7200 48GB Kit 견적에 담아줘`를 전송했다.
3. 챗봇은 `현재 견적에 같은 RAM이 이미 들어가 있어요. 추가로 몇 개를 넣을까요?`라고 답했다.
4. `1개`라고 답하자 `RAM 조건에 맞는 내부 자산 후보를 찾지 못했습니다.`로 끝났다.

**네트워크 대조**

- `currentQuoteDraft`에는 RAM이 없었다.
- `currentBuilds`에는 이전 GPU 변경 미리보기와 함께 제거 전 RAM이 남아 있었다.
- 후속 `1개` 요청은 `needsDraftContext`에 걸리지 않아 최신 `currentQuoteDraft` 자체가 전송되지 않았다.

**기대 결과**

- 현재 견적 사실의 우선순위는 항상 `서버의 활성 quote draft > 아직 적용하지 않은 과거 preview`여야 한다.
- 되묻기 후속 답변은 원문의 intent와 draft 필요 여부를 상속해야 한다.
- preview는 추천 참고 정보일 뿐 현재 장착 여부 판단에 사용하면 안 된다.

**추정 원인**

- 프론트는 모든 요청에 최근 `currentBuilds`를 보내지만 최신 draft는 정규식이 인식한 문장에만 보낸다.
- `1개` 같은 슬롯 채우기 응답은 독립 문장만으로 draft 요청인지 판정할 수 없다.
- 서버/LLM 입력 계약에 authoritative state와 speculative preview의 우선순위가 명시적으로 분리되어 있지 않다.

관련 코드: `AiBuildAssistant.tsx:336`, `AiBuildAssistant.tsx:344`, `AiBuildAssistant.tsx:827`.

### P1-4. AS 실제 로그 근거가 발표 화면에 나타나지 않음

**재현**

- fixture에는 Display 이벤트 4101과 `nvlddmkm` 반복 신호가 있다.
- DB의 `supportRouting.reasonCodes`에는 `DRIVER_ERROR_REPEAT`가 저장됐다.
- 사용자 화면에서는 `Driver and OS errors are remote-support first cases`라는 일반 영문 사유만 표시됐다.
- 관리자 화면도 raw JSON과 code 중심이라 발표자가 즉시 읽을 수 있는 근거 행이 없다.

**기대 결과**

### P2-4. 홈 추천 조합 가격이 비현실적임

### P2-5. 사용자 문구 세부 품질

## 5. 통과한 핵심 방어

- TARGET 예산 요청의 모든 추천 카드가 ±12.5% 범위를 지켰다.
- GPU 하향 미리보기는 가격·FPS·종합 점수 변화를 표시하고 자동 저장하지 않았다.
- 정확 상품명 복구 미리보기와 적용이 성공했다.
- 5090 FPS 및 9700X CPU 시뮬레이션은 `actions=[]`이고 견적을 바꾸지 않았다.
- 단일·복수 부품 위치 질문은 URL과 견적을 바꾸지 않고 배치도 spotlight만 변경했다.
- 모호한 `게임하고 싶어`, `컴퓨터 맞춰줘`는 예산/용도를 되묻고 재사용 가능한 칩을 제공했다.
- 조립 중개에서 내부·외부 제안, 개인정보 단계 공개, 가상 결제가 실제 API로 동작했다.
- 수동 견적은 빈 상태에서 호환 조합 8/8까지 완성할 수 있었다.
- 원격지원 요청과 관리자 처리 상태는 사용자 화면에 polling으로 반영됐다.

## 6. 기존 대량 QA가 놓친 이유

기존 200건 시나리오 보고서는 개별 요청 기준으로 schema/action/route와 draft 무변조를 검사했다. 700건 부하 검증도 API-level 독립 입력과 응답 유효성을 중심으로 했다. 이들은 넓은 문장 다양성과 동시 부하에는 유효하지만 다음을 오라클로 두지 않았다.

1. 추천 카드의 라벨과 이후 BuildGraph 재평가가 같은 사실을 말하는가.
2. preview 적용 또는 삭제 후 다음 턴이 최신 draft를 읽는가.
3. quick reply를 다시 보냈을 때 이전 문제를 실제로 개선하는가.
4. API/DB에 있는 근거가 사용자 화면에 이해 가능한 형태로 나타나는가.
5. 사용자, 기사, 관리자 계정의 상태 전이가 서로 맞물리는가.
6. 사용자가 클릭할 수 있는 버튼과 실제 서버 action이 끝까지 성공하는가.

즉, 기존 QA의 `한 요청 → 한 응답` 단위가 통과해도 `응답 → 적용 → 다음 요청`의 경계에서 오류가 발생할 수 있었다.

## 7. QA 체계 개편안

### 7.1 상태 전이 시나리오를 1급 테스트로 관리

각 시나리오는 다음 필드를 가져야 한다.

- 시작 계정과 DB fixture
- 시작 quote draft fingerprint
- 사용자 입력과 클릭 동작
- 기대 UI 표시
- 기대 API payload
- 기대 DB 변화 또는 무변화
- 다음 턴에 전달할 authoritative context
- 종료 상태와 원복 절차

### 7.2 세 종류의 오라클을 동시에 사용

| 오라클 | 검증 내용 |
|---|---|
| 기능 오라클 | route, simulation, preview, apply, 결제, 지원 상태가 맞는가 |
| 사실 오라클 | 가격, 점수, Tool 결과, 로그 근거가 DB/공통 평가와 일치하는가 |
| 대화 오라클 | 후속 문장이 이전 턴과 현재 상태를 올바르게 이어받는가 |

### 7.3 반드시 고정할 불변식

- 현재 견적 판단: 서버 활성 draft가 preview와 채팅 history보다 우선한다.
- 읽기 전용 intent: simulation, 설명, 위치 강조는 DB mutation 0건이다.
- 추천 후속: `상위/더 여유/더 빠름`은 실제 score 또는 측정값이 개선돼야 한다.
- 교차 화면 정합성: 추천 카드, BuildGraph, 점수 설명의 Tool status와 warning code가 일치한다.
- 증거 표시: 발표 문구로 약속한 근거는 UI DOM에서 실제 확인 가능해야 한다.
- quick reply 재사용: 칩 전송 후 dead-end, 동일 문제 반복, 무의미한 동급 교체가 없어야 한다.

### 7.4 최소 회귀 묶음

1. 200만원 QHD 추천 → 가운데 적용 → BuildGraph 재평가 일치.
2. GPU 하향 → 적용 → 정확 원복 → score 원복.
3. RAM 제거 → 같은 RAM 담기 → 수량 응답 `1개` → preview/apply.
4. 점수 약점 설명 → 우선순위 칩 → 실제 개선 후보의 delta 양수.
5. 조립 요청 → 외부 입찰 → 사용자 polling → 선택 → 결제 → 기사 개인정보 공개.
6. driver fixture 업로드 → 사용자 evidence row → 관리자 승인 → 원격지원 갱신.

## 8. 수정 우선순위

1. **상태 권위 정리 — 완료**: 셀프견적의 모든 요청과 clarification 후속에 최신 `currentQuoteDraft`를 첨부한다. `currentBuilds`는 대화용 speculative preview로만 남기고 Tool·후보 검증은 활성 draft를 기준으로 한다.
2. **공통 평가 결과 사용**: 추천 카드의 tier, Tool badge, 종합 점수 설명을 같은 평가 snapshot에서 생성한다.
3. **실질 개선 guard**: 상위/여유/속도 추천은 현재 대비 category score 또는 핵심 측정값이 증가하지 않으면 서빙하지 않는다.
4. **AS evidence presenter**: reason code와 마스킹된 raw sample을 사용자용 한글 근거 행으로 변환한다.
5. **동적 문구 정리**: build 개수, 최소 구성 카드, 한국어 조사와 언어 오염을 응답 후처리에서 검증한다.
6. **발표 전 상태 기반 E2E**: 위 최소 회귀 묶음을 실제 Docker와 seed 계정에서 매번 실행한다.
